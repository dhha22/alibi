alibi.

git blame은 누가를 말한다. alibi는 왜를 말한다.

경력 20년, 검거율 0%의 강력계 형사. 수상한 코드 한 줄을 짚어 주면 — blame이 자꾸 엉뚱한 사람에게 뒤집어씌우는 prettier 커밋들을 걷어내고, 그 줄의 의미를 실제로 작성한 기원 커밋까지 파 내려가, PR·이슈·리뷰 스레드를 추적해 코드의 알리바이를 판결합니다.

사건 접수 → 세 가지 판결 사건 파일 보기
EXHIBIT A — 증거판 앞의 형사. 핀으로 꽂힌 카드 하나하나가 커밋이고, 실 한 가닥 한 가닥이 단서다.
“범인을 데려오지 마. 이야기를 가져와.”

FILE 01형사 소개

personnel file — homicide div.
경력강력계 20년
검거율0%. 의도된 기록이다
전문 분야코드가 왜 거기 있는지 규명 — 누구 탓인지가 아니라
권한없음. 사실만 확정한다 — 처분은 당신의 몫

신입 시절, 선배의 어긋난 rebase가 그가 쓴 적 없는 심야 핫픽스에 그의 이름을 남겼고 — 장애 회고에서 추궁당한 건 그 혼자였다. 그날 이후 그는 blame이 가리키는 이름을 믿지 않는다. 수상한 코드를 보면 “누가 이랬어”라고 묻지 않는다 — “여기서 무슨 일이 있었나”라고 묻는다.

FILE 02blame은 엉뚱한 증인을 지목한다

몇 년쯤 살아남은 코드에 맨 git blame을 돌려 보면, 마지막으로 prettier를 돌린 사람 이름이 나온다. 진짜 작성자와 진짜 이유는 포매팅 스윕, 린트 수정, rename, 파일 이동 밑에 파묻혀 있다. alibi는 노이즈 판별 루프를 돌린다: blame이 지목한 커밋을 심문하고, 기계적 변경임을 입증하고, 제외하고, 다시 blame — 그 줄의 의미를 작성한 커밋에 닿을 때까지.

✕ blame의 진술

$ git blame db/client.ts -L 88,88 a1b2c3d (@dohyun 2023) setTimeout(resolve, 0) → "style: apply prettier" — 412개 파일 변경. @dohyun은 이 줄을 쓴 적이 없다.

보고서의 이름은 포매팅 커밋의 것. @dohyun에게 이 줄이 왜 있냐고 물어보면 두 사람의 오후만 날아간다.

○ alibi의 수사 결과

$ blame -w --ignore-rev a1b2c3d … 4f2a91c (@minji 2021) setTimeout(resolve, 0) → "fix: safari indexeddb tx race" · PR #482 리뷰어: "WebKit이 고치면 제거할 것"

기원 커밋 확보, PR 추적, 리뷰어의 약속까지 회수 — 이제 “지워도 되나요?”라는 질문이 답할 수 있는 모양이 된다.

FILE 03모든 사건은 세 가지 판결 중 하나로 끝난다

이 스킬은 체스터턴의 울타리 판독기다. 질문은 언제나 “이 코드 이상해 보이는데 — 지워도 되나요?”의 변형이고, 답은 얼버무림 속에 숨지 않는다. 보고서의 모든 주장에는 증거물(E-n) 또는 증인 진술(W-n)이 붙는다. 증거 없으면 주장도 없다.

⚖️

JUSTIFIED

이유는 실재했고 — 지금도 유효하다. 우회하던 버그가 아직 살아 있고, 지원하던 플랫폼이 아직 지원된다.

→ 건드리지 마라. 울타리엔 목적이 있다.
⌛

EXPIRED

이유는 실재했지만 이미 죽었다 — 업스트림에서 버그가 고쳐졌고, 플랫폼 지원이 끊겼고, 마이그레이션이 끝났다.

→ 제거해도 안전. 주의사항 첨부.
🕳️

COLD CASE

이유를 찾을 수 없었다 — 기록 유실, 구두 결정, squash된 히스토리. alibi는 사건을 닫으려고 그럴듯한 이야기를 지어내지 않는다.

→ 최대한 신중히. 마지막 증인 명단 첨부.
🕊️ 사람은 절대 용의자가 아니다. 사건 파일에 등장하는 모든 사람은 증인이다 — 비난의 표적이 아니라. 이 스킬의 존재 이유가 blame 문화로부터의 탈출이다: “@minji는 용의자가 아닙니다. 이 사건의 핵심 증인입니다.”

FILE 04여섯 가지 수사 모드

형사는 한 명, 사건 유형은 여섯. 질문의 모양을 보고 스스로 모드를 고른다 — 당신이 모드 이름을 알 필요는 없다.

모드당신이 가져오는 질문서에서 나가는 것
why-not-who“이 줄은 왜 존재하나 — 지워도 되나?”판결이 붙은 사건 파일
commit-to-story“이 커밋/PR은 왜 들어갔나 — 무슨 논쟁이 있었나?”조서: 문제 → 논쟁 → 결정 → 후일담
first-broken“전엔 됐는데 — 어느 커밋이 깨뜨렸나?”검증된 프로브로 bisect → 범인 커밋 심문
incident-trace“X에 무슨 일이 있었나 — revert됐나? 어느 릴리스에 탔나?”사건 타임라인 + 릴리스별 노출 범위
design-rationale“왜 이렇게 설계됐나 — 다른 시도는 없었나?”판례 기록: 시대 구분 + 기각된 대안들
repo-tour“이 리포의 역사로 온보딩해 줘 — 시체는 어디 묻혀 있나?”다섯 장면 관할구역 투어 + 첫수 조언

FILE 05종결된 사건 파일의 모습

FILE 02의 수수께끼 setTimeout(resolve, 0), 종결편. 증거물, 증인 진술, 추론 체인, 판결 — 그리고 처분은 끝까지 당신의 몫이다.

∴  추론

   ∴  Verdict: EXPIRED

FILE 06형사를 부르는 법

주문은 필요 없다 — 어차피 당신이 이미 하고 있던 질문들이다. 사건 파일은 당신이 말한 언어로 돌아온다.

이 타임아웃 왜 30초야? 이 setTimeout 지워도 돼? 의미 없어 보이는데 이거 누가 왜 썼어 — prettier 커밋 말고 커밋 4f2a91c의 전체 스토리 알려줘 지난주엔 됐는데 — 깨뜨린 커밋 찾아줘 이거 revert된 적 있어? 어느 릴리스에 탔어? 왜 Promise.all이 아니라 큐로 짰지? 이 리포, 시체 어디 묻혀 있어? /alibi src/db/client.ts:88 run alibi
